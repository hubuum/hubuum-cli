# Design: Adapt hubuum-cli to hubuum_client 0.0.3

- **Date:** 2026-06-28
- **Branch:** `feat/client-0.0.3`
- **Status:** Approved design, pending spec review

## Goal

Prepare the CLI to support the next iteration of the client library
(`hubuum_client`, locally at `../hubuum-client-rust`, version `0.0.3`). The CLI
currently depends on `hubuum_client = "0.0.2"` (crates.io). Version `0.0.3`
introduces large breaking changes:

- A completely reworked, **principal-centric IAM** (users + service accounts as
  principals; `me/*` self-serve identity; principal tokens; principal namespace
  permissions; expanded `Permissions` enum).
- **Async, task-based reports** (`submit` ŌåÆ poll ŌåÆ `output`) replacing the
  previous synchronous flow.
- A general **tasks** subsystem (`tasks().get/events/wait/query`).
- Task-based **imports**.
- New **remote targets** (CRUD + invoke, task-backed).
- Changed task types (`TaskResponse.links: Option<ŌĆ”>`, `TaskDetails`, new
  `TaskKind` variants `Reindex`/`RemoteCall`, report links).
- Unified search batch/stream changes.

This work happens on a dedicated, temporary test branch so the new library can be
exercised against an updated server before `0.0.3` is published.

**Scope:** full surface ŌĆö compile-to-parity for existing commands, the async
reports/tasks rework, the new IAM surface, and the other new areas (remote
targets, report `include`/`relation_context`, unified search stream).

**Test target:** compile-only for now plus a documented manual-test checklist; a
live smoke run happens later, by the user.

## Approach

Single branch, adapted in ordered phases. Each phase compiles cleanly before the
next begins, so breakage stays isolated and reviewable. (Rejected: a big-bang
rewrite that compiles only at the end ŌĆö too hard to debug across IAM + reports +
tasks at once; and a two-branch split ŌĆö the user wants one branch.)

## Core design principle: unified task ergonomics

Reports, remote-call/invoke, imports, and any future task kind all produce a
server-side task. They must share one consistent set of ergonomics rather than
each inventing its own. The CLI gains a thin shared layer over the library's
per-domain task APIs:

### Submit flow (shared helper)

Every command that creates a task is a **background submit by default**; all of
them support `--wait` to block instead. A single command-layer helper wraps every
task-producing operation:

1. Call the domain submit (`reports().submit`, `remote_targets().invoke`,
   `imports().submit`), normalizing the returned `TaskResponse` into the existing
   `TaskRecord` domain type.
2. Register the task with `BackgroundManager` (already kind-agnostic) with a
   consistent label `"<kind> #<id>"`.
3. Default (no `--wait`): print `submitted task #<id> (<kind>)` and return
   immediately, leaving the background poller to track it. The user later inspects
   it and views results via the `jobs` command (below).
4. With `--wait`: poll to terminal inline (with progress), then fetch and render
   output directly.

This default is **universal** ŌĆö there is no per-command exception; if a command
creates a task, it submits in the background and accepts `--wait`.

Shared flags on every submitter: `--wait`, `--timeout`, `--poll-interval`,
`--idempotency-key`.

### `jobs` command group (local view: submitted jobs + results)

`jobs` is the user-facing entry point for everything submitted in the background.
It supersedes the current `bg` command group (`bg list/show/watch/forget`), which
is renamed to `jobs` (a `bg` alias may be kept for muscle memory). Because
`BackgroundManager` is kind-agnostic, reports, imports, and remote calls all show
up here uniformly.

- `jobs` (bare) / `jobs list` ŌĆö list tracked jobs with relevant details: local
  id, task id, kind, state (running / done / failed / attention), progress, and
  a short summary, so the user can see at a glance what is outstanding and what
  finished.
- `jobs show <local-id>` ŌĆö full detail for one job, including how to view its
  result.
- `jobs output <local-id>` ŌĆö render the task's result (delegates to the unified
  output abstraction, dispatched by `TaskKind`).
- `jobs watch <task-id>` ŌĆö start tracking an existing server task.
- `jobs forget <local-id>` ŌĆö stop tracking a job locally.

### Generic `task` command group (direct server-task access, all kinds)

For working with server tasks by their server-side id (independent of local
tracking):

- `task list [--kind <k>] [--status <s>]` ŌĆö backed by `tasks().query()`.
- `task show <id>` ŌĆö `tasks().get()`.
- `task events <id>` ŌĆö `tasks().events()`.
- `task wait <id> [--timeout] [--poll-interval]` ŌĆö `tasks().wait()`.
- `task output <id>` ŌĆö dispatches output retrieval by the task's `TaskKind`.
- `task queue` ŌĆö existing queue-state view (`meta_tasks`).

Domain wrappers (`report output <id>`, `import results <id>`) and `jobs output`
all delegate to the same machinery so behavior is identical regardless of entry
point.

### Unified output abstraction

`HubuumGateway` exposes `task_output(task_id) -> TaskOutput`, where `TaskOutput`
is a normalized enum:

```
enum TaskOutput {
    Report(ReportResult),          // Json | Rendered
    ImportResults(Vec<ImportTaskResultResponse>),
    RemoteCall(RemoteCallResult),  // shape per lib
    None,                          // task kind without retrievable output
}
```

The gateway inspects the fetched `TaskResponse.kind` to choose the right
library call. Formatting lives in `src/formatting/` with one renderer per
variant.

### Prompt / status (single source)

`BackgroundManager` already tracks `TaskRecord`s of any kind and produces a
transient badge (`[bg:N done:M attention:K]`) consumed in `app.rs::prompt()`.
We extend it with a **persistent** outstanding-job indicator so the prompt always
reflects current state, not just one-shot notices ŌĆö e.g.
`[jobs 2Ō¢Ė 1Ō£Ś]` (running / failed) rendered before the existing transient notices.
Because the manager is kind-agnostic, reports, imports, and remote calls all
appear uniformly.

## Phases

### Phase 0 ŌĆö Branch & dependency wiring

- Create branch `feat/client-0.0.3` off `main`.
- Commit `hubuum_client = "0.0.3"` in `Cargo.toml` (correct once published).
- Add a **local, uncommitted** override: gitignored `.cargo/config.toml` with
  `paths = ["../hubuum-client-rust"]`. The `paths` override requires a matching
  semver, which is why the committed `0.0.3` bump is needed.
- Add `.cargo/config.toml` to `.gitignore`.
- Verify `cargo build` resolves the local path (it will not compile yet).

Reverting later: publish `0.0.3`, delete the gitignored `.cargo/config.toml`.
No tracked-file change is needed to undo the override.

### Phase 1 ŌĆö Compile to parity

Reconcile breaking changes module-by-module until the existing command set
compiles and behaves as before. Affected areas (exact per-symbol deltas are
enumerated in the implementation plan):

- `src/background.rs`, `src/domain/tasks.rs`, `src/services/gateway/tasks.rs`:
  `TaskResponse` (`links: Option<ŌĆ”>`, `details: Option<TaskDetails>`, removed
  fields), `TaskLinks` (new `report`/`report_output`), `TaskKind`
  (`Reindex`/`RemoteCall`).
- `src/services/gateway/reports.rs`, `src/domain/reports.rs`,
  `src/commands/report.rs`: move to the `reports()` handle; report templates
  become a `Resource<ReportTemplate>` (replacing `ReportTemplatePost/Patch`
  usage with the resource builders).
- `src/services/gateway/{users,groups,namespaces}.rs` and domains: expanded
  `Permissions` enum and new permission grant/replace/revoke signatures;
  principal-centric user/group shapes.
- `src/services/gateway/imports.rs`, `src/domain/imports.rs`: task-based result
  shape (`ImportTaskResultResponse`).
- `src/services/gateway/search.rs`: batch/kind signature changes.
- `src/app.rs`: auth (`login`/`login_with_token`/`get_token`/logout) ŌĆö verify
  signatures, adjust if changed.

Deliverable: `cargo build` and `cargo test` green; existing commands unchanged in
behavior.

### Phase 2 ŌĆö Async reports + unified task ergonomics + prompt status

- Implement the shared submit helper, the `jobs` and generic `task` command
  groups, the `TaskOutput` abstraction, and the shared flags described above.
- Rename the `bg` command group to `jobs` (keep `bg` as an alias) and extend it
  with `jobs output` and richer `jobs list` details.
- Make all task-creating commands background-submit-by-default with `--wait`,
  starting with `report run` and imports (remote-call follows in Phase 3).
- Extend `BackgroundManager` with persistent outstanding-job status and update
  `app.rs::prompt()` rendering.

Deliverable: reports/imports run through one consistent task pipeline (background
by default, `--wait` to block); `jobs` lists and renders results; prompt shows
live job status; `cargo build`/`cargo test` green.

### Phase 3 ŌĆö New full-surface commands

- **IAM:**
  - `service-account` CRUD + token management (create/list/revoke), `disable`.
  - `me` / `whoami`: identity (`me`), `me groups`, `me tokens`, `me permissions`.
  - Principal token management for users and service accounts
    (`NewTokenRequest` builder; create/list/revoke).
  - Principal namespace permissions (view; grant/revoke via namespace).
  - Wire the expanded `Permissions` enum (template + remote-target variants) into
    permission parsing/printing.
- **Remote targets:** `remote-target` CRUD + `invoke` (task-backed, via the
  shared submit helper).
- **Reports:** expose `include` and `relation_context` options on `report run`.
- **Search:** unified search `stream` mode.

Deliverable: new command areas available; `cargo build`/`cargo test` green.

### Phase 4 ŌĆö Manual-test checklist

- Write `docs/manual-test-0.0.3.md`: an ordered checklist of commands to exercise
  each area (auth, classes/objects, relations, search + stream, reports
  sync/`--wait`/background + output, imports, tasks, service accounts, me/*,
  principal tokens & permissions, remote targets) against an updated server,
  with expected outcomes.

## Out of scope

- Committing the path override (`.cargo/config.toml` stays gitignored).
- Automated integration tests against a live server.
- Unrelated refactoring beyond what each touched module needs to compile cleanly
  and fit the unified task pattern.

## Risks / open points

- **API churn:** the local lib is still in development; signatures may shift.
  Phasing + compile-after-each-phase contains the blast radius.
- **Output shapes:** exact `RemoteCallResult` / report `Rendered` rendering is
  confirmed against the lib during Phase 2/3.
- **`paths` override semantics:** requires matching semver; mitigated by the
  committed `0.0.3` bump. If the override misbehaves, fall back to an
  uncommitted inline `Cargo.toml` path edit (left unstaged).
