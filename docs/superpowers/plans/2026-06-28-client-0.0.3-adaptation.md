# hubuum_client 0.0.3 Adaptation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Adapt hubuum-cli to compile and run against `hubuum_client` 0.0.3 (full surface), with unified ergonomics for all task-backed operations, a `jobs` command, persistent prompt status, and the new IAM / remote-target / search-stream surfaces.

**Architecture:** One branch (`feat/client-0.0.3`), adapted in ordered phases that each compile before the next. The dependency is redirected to the local lib via a gitignored `.cargo/config.toml` `paths` override plus a committed `0.0.3` version bump. All task-creating commands submit in the background by default (with `--wait` to block), share one submit helper, and surface through one `jobs` command and one generic `task` command group.

**Tech Stack:** Rust 2021/2024, `hubuum_client` 0.0.3 (sync client), `tokio` runtime, `cli_command_derive` (`#[derive(CommandArgs)]`), custom command catalog.

## Global Constraints

- Dependency: committed `hubuum_client = "0.0.3"`; local override lives ONLY in gitignored `.cargo/config.toml` — never commit a path dependency in `Cargo.toml`.
- Every command that creates a server task is background-submit-by-default and MUST accept `--wait` (alias for block-and-render). No per-command exceptions.
- All gateway methods return `Result<_, AppError>`; never leak `hubuum_client::ApiError` past the gateway layer (map via existing `errors.rs`).
- Domain records wrap `hubuum_client` types (`transparent_record!`) or are resolved custom structs; commands talk to `services.gateway()` / `services.background()`, never to `hubuum_client` directly.
- New commands follow the existing pattern in `src/commands/imports.rs`: `#[derive(CommandArgs)]` struct with `#[option(...)]`, `impl CliCommand`, registered via `catalog_command` in a `register_commands`.
- Run `cargo build` and `cargo test` after every task; both must be green before commit. **Phase 1 exception (user-approved):** tasks 1.1–1.6 fix one module each and may leave the workspace non-compiling between tasks — the deliverable for those tasks is "the named module's errors resolved + existing tests still pass", and a fully green workspace is required only at the 1.7 parity gate and throughout Phases 2–4. New unit tests are NOT required for Phase 1's pure API-reconciliation edits (the compiler is the test); Phases 2–4 follow normal TDD.
- Commit after every task. Co-author trailer: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.

## Reference: confirmed API facts

These were extracted from the local lib and current CLI and are used throughout the plan.

Auth (0.0.3): `Credentials::new(name: String, password: String)` (fields private, `username`→`name`); `Token::new(token: String)`; `BaseUrl: FromStr`; `client.get_token() -> &str`; `client.logout()/logout_token(&str)/logout_user(i32)/logout_all()`.

Tasks (0.0.3): `client.tasks() -> Tasks` with `get(i32)`, `events(i32) -> CursorRequest<TaskEventResponse>`, `wait(i32) -> TaskWaitOp` (`.poll_interval(Duration).timeout(Option<Duration>).send()`), `query() -> TaskListRequest` (`.kind/.status/.submitted_by/.limit/.sort/.cursor/.page()/.list()`). `TaskResponse.links: TaskLinks` (NOT Option); `TaskLinks` adds `report: Option<String>`, `report_output: Option<String>` to existing `task,events,import_url,import_results`. `TaskResponse.details: Option<TaskDetails>` with `import_details: Option<ImportTaskDetails{results_url}>`, `report: Option<ReportTaskDetails>`. `TaskStatus::{Queued,Validating,Running,Succeeded,Failed,PartiallySucceeded,Cancelled}` with `.is_terminal()/.is_success()`. `TaskKind::{Import,Report,Export,Reindex,RemoteCall}`.

Reports (0.0.3): `client.reports() -> Reports` with `submit(ReportRequest) -> ReportSubmitOp`, `run(ReportRequest) -> ReportRunOp`, `get(i32)`, `output(i32) -> ReportResult`. `ReportResult::{Json(ReportJsonResponse), Rendered{content_type, body}}`. Templates via `client.templates() -> Resource<ReportTemplate>`.

IAM (0.0.3): `users()/service_accounts()/groups() -> Resource<T>`. `me()/me_groups()/me_tokens()/me_permissions()`. Handle methods: `SyncHandle<User>::{groups,tokens,tokens_create(NewTokenRequest)->String,token_revoke(i32),set_password(impl Into<String>)}`; `SyncHandle<ServiceAccount>::{disable,tokens,tokens_create,token_revoke}`; `SyncHandle<Group>::{add_member(i32),remove_member(i32),members()->Vec<PrincipalMember>}`. `Permissions` enum: 29 variants (adds `ReadTemplate,CreateTemplate,UpdateTemplate,DeleteTemplate,ReadRemoteTarget,CreateRemoteTarget,UpdateRemoteTarget,DeleteRemoteTarget,ExecuteRemoteTarget`).

Namespaces (0.0.3): `SyncHandle<Namespace>::{grant_permission(group_id,Permissions), revoke_permission, grant_permissions(group_id,Vec<String>), replace_permissions, revoke_permissions(group_id), group_permissions(group_id)->PermissionResult, permissions()->Vec<GroupPermissionsResult>, principal_permissions(principal_id), groups_with_permission(Permissions)}`.

Remote targets (0.0.3): `client.remote_targets() -> Resource<RemoteTarget>`; `SyncHandle<RemoteTarget>::invoke(RemoteTargetInvokeRequest) -> TaskResponse`. `RemoteTargetInvokeRequest::new(RemoteInvocationSubject).parameters(Value).body_override(Value)`. Result fetched as `RemoteCallResult`.

Search (0.0.3): `client.search(query) -> UnifiedSearchRequest` with `.kinds/.limit_per_kind/.cursor_*/.search_class_schema/.search_object_data/.execute()->UnifiedSearchResponse/.stream()->Vec<UnifiedSearchEvent>`.

Imports (0.0.3): `client.imports() -> Imports` with `submit(ImportRequest)->ImportSubmitOp`, `get(i32)`, `results(i32)->CursorRequest<ImportTaskResultResponse>`. `ImportRequest{version:i32, dry_run:Option<bool>, mode:Option<ImportMode>, graph:ImportGraph}`; `ImportMode` is now a struct (`atomicity/collision_policy/permission_policy`), not an enum; `ImportGraph` uses non-Option `Vec` fields incl. `namespace_permissions`. `ImportTaskResultResponse{id,task_id,item_ref,entity_kind,action,identifier,outcome,error,details,created_at}`.

Resource<T> CRUD: `find()/query() -> QueryOp<T>` (`.add_filter_*`, `.list()/.one()/.optional()/.page()`), `create() -> CreateOp<T>` (`.params(T::PostParams).send()`), `update(id) -> UpdateOp<T>` (`.params(T::PatchParams).send()`), `delete(id)`, `select(id) -> Handle<T>`, `select_by_name(name) -> Handle<T>`.

> **Lookups to confirm at implementation time** (concrete, not design decisions — read the lib's `ApiResource` impls): the associated `PostParams`/`PatchParams`/`GetOutput` types for `User`, `ServiceAccount`, `Group`, `Namespace`, `ReportTemplate`, `RemoteTarget` (e.g. `NewRemoteTarget`/`UpdateRemoteTarget` are already confirmed for `RemoteTarget`). Where a task below says "build `T::PostParams`", read the exact struct from the lib first.

```rust
#[derive(Debug, Clone, Default)]
pub struct TaskSubmitOptions {
    pub wait: bool,
    pub timeout_secs: Option<u64>,
    pub poll_interval_secs: Option<u64>,
}

pub fn run_task_backed(
    services: &AppServices,
    tokens: &CommandTokenizer,
    label: impl Into<String>,
    opts: TaskSubmitOptions,
    task: TaskRecord,
) -> Result<(), AppError> {
    let label = label.into();
    let task_id = task.0.id;
    let kind = task.0.kind;

    if opts.wait {
        let final_task = services.gateway().wait_task(WaitTaskInput {
            task_id,
            timeout_secs: opts.timeout_secs,
            poll_interval_secs: opts.poll_interval_secs,
        })?;
        let output = services.gateway().task_output(task_id)?;
        match desired_format(tokens) {
            OutputFormat::Json => append_line(serde_json::to_string_pretty(&output)?)?,
            OutputFormat::Text => {
                final_task.format_noreturn()?;
                for line in output.render_lines() { append_line(line)?; }
            }
        }
        return Ok(());
    }

    let registration = services.background().watch_task(task.clone(), label);
    match desired_format(tokens) {
        OutputFormat::Json => append_line(serde_json::to_string_pretty(&task)?)?,
        OutputFormat::Text => {
            append_line(format!("submitted task #{task_id} ({kind})"))?;
            if let Some(reg) = registration {
                append_line(format!("tracking as background job {}", reg.local_id))?;
            }
        }
    }
    Ok(())
}
```

```rust
pub fn prompt_status(&self) -> Option<String> {
    let guard = self.inner.lock().ok()?;
    if !guard.enabled { return None; }
    let running = guard.jobs.values().filter(|j| j.poller_running).count();
    let failed = guard.jobs.values().filter(|j|
        j.task.as_ref().map(|t| is_attention_status(t.0.status)).unwrap_or(false)
        || j.last_error.is_some()).count();
    let done = guard.jobs.values().filter(|j|
        j.task.as_ref().map(|t| is_terminal_status(t.0.status) && !is_attention_status(t.0.status)).unwrap_or(false)).count();
    if running == 0 && failed == 0 && done == 0 { return None; }
    let mut parts = Vec::new();
    if running > 0 { parts.push(format!("{running}▸")); }
    if done > 0 { parts.push(format!("{done}✓")); }
    if failed > 0 { parts.push(format!("{failed}✗")); }
    Some(format!("[jobs {}]", parts.join(" ")))
}
```
